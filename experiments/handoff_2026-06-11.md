## 铁律（吃过亏，务必遵守）
1. **实车（人类 Webots + 肉眼）是唯一裁判**；离线指标只用来提假设/抓全局回归，**绝不能否决实车结论**。
2. **`lost` 率不是质量指标**：车 lost 时直线滑行（不停车、不偏出），lost 多≠开得差。当前 lost≈0.53 主要是 gate-bridge 把含天空帧判饱和→lost，是**良性虚高**，但它**确实拖慢速度**（lost→`lost_speed` 0.24）。
3. 任何感知改动要用整场记录找关键窗口复盘，不能只看目标点（局部修好全局砸锅出现过两次）。
4. 用领域常识 sanity check：**basic 是顺时针、全程右弯**（曲率应恒为正；出现"满信心左弯"必是 bug）。
5. policy/速度/走线类改动**没有离线指标**，只能让人上车判断；**不要 autonomous 盲调**。感知类改动可离线筛（见 `docs/ai_offline_review.md`），但仍需人上车终判。
6. 调试构建（含 `open/json/np.save`）**禁上传**；提交只用 `submissions/final/team_controller.py`，必须过 `validate_submission.py` + 官方 validator + `pytest`。

## 未解问题（按优先级 + 诊断）

### P1. 直道太慢（机制对、欠触发）
- 现状：`straight_speed`（basic 0.78）确实够到（max speed=0.780），但**只命中 24% 帧**。两个原因：
  1. **53% 帧 lost → 限速 0.24**（即使在直道上）。
  2. **直道条件太严**：`curve_risk=max(|curv|,|head|,|look|)`，其中 `heading` 噪声大（非lost帧 mean=0.13>阈值0.12）→ `curve_risk` 常 >0.12 → 判不成直道。
- 方向（需人上车验证每一步）：
  - 放宽/稳健化"直道判定"：把 `straight_curve_max` 调高，或直道性改用更稳的信号（如连续 N 帧低曲率、或主要看 `curvature/lookahead` 而非噪声大的 `heading`）。
  - **处理 lost 帧的限速**：车 lost 但"最近一直在直道高置信"时，不该立刻砸到 `lost_speed` 0.24——可做"惯性滑行/最近直道记忆"，让短暂 lost 时保持速度。这是提速最大杠杆（53% 帧）。
  - 可能再提 `straight_speed` 和加速率（`max_speed_increase_per_sec`）。
- 参数都在 `params.py`（`straight_speed/straight_curve_max/straight_offset_max`、basic override）。

### P2. "第二个弯"提前拐 + 贴右(内)栏杆
- **更正**：不是 Y 分叉，是**单纯右弯**，但**车道白线画成 3 段不连续**，接缝处乱。
- 推测：感知中心线在白线接缝处被带偏 → 近处 `lateral/heading` 跳变 → 骗开入弯门控（门控正是靠这俩判断"弯到没到"）→ 早拐 → 切内线贴右栏。
- **用户诉求**：让车尽量**保持在白线上/跟中心线**，别脱出贴栏杆。
- 下一步：用 `frames_C006` 按 `docs/ai_offline_review.md` 的流程，在控制日志里定位该弯（找 `lateral/heading` 在接缝处跳变的段），调出那几帧的摄像头、mask、中心点 overlay，看是哪段线/接缝把中心线带偏，再对症修感知（estimator/perception），**别只调门控**（门控信号本身被污染了）。

### P3. lost≈0.53 偏高（良性但拖速）
- 来自 gate-bridge（蓝门并入 mask 时部分含天空帧饱和→判 lost）。**实车更好**（蓝门不再乱拐），所以**不要为降这个数字而回退 gate-bridge**（铁律 #2，已在 R005 栽过）。
- 若要降它，只能在**不破坏实车表现**前提下，让"含天空的饱和帧"不被误判 lost——属感知，谨慎，全局验证。它和 P1 的 lost 限速是同一痛点的两面。
